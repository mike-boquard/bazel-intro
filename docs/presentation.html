Why Bazel?

You already have make, cmake, go build, cargo…


Hermetic

Builds don't depend on your environment — same inputs → same outputs, always.

Incremental

Tracks changes by content hash, not timestamps. Only rebuilds what actually changed.

Polyglot

C++, Go, Java, Python, Proto — one build graph, one command to run them all.


The caching story alone pays off on large codebases. CI restores the same cache that dev machines write.

What "hermetic" means in practice

Traditional build

  • Reads from /usr/include
  • Depends on installed gcc version
  • "Works on my machine"
  • Rebuild triggers unpredictable

Bazel build

  • All deps declared explicitly
  • Toolchain fetched and pinned
  • Sandboxed — no ambient env
  • Cache key = content hash of inputs

Key vocabulary

Workspace — the repo root; contains MODULE.bazel
  └─ Package — any directory with a BUILD.bazel
     └─ Target — a named buildable thing (//cpp/greeter:greeter)
         └─ Rule — the type of target (cc_library, go_binary, …)

A label like //cpp/greeter:greeter_test means:
  //cpp/greeter = package path from workspace root
  :greeter_test = target name within that package

Bzlmod & MODULE.bazel

Bazel's modern dependency system — replaces the legacy WORKSPACE file.


# MODULE.bazel — every external dep declared here

module(name = "bazel_intro", version = "0.0.1")

# ── BCR deps ───────────────────────────────────────────────
bazel_dep(name = "googletest",  version = "1.15.2")
bazel_dep(name = "abseil-cpp",  version = "20240722.0")
bazel_dep(name = "rules_go",    version = "0.50.1")
bazel_dep(name = "protobuf",    version = "29.3")
bazel_dep(name = "grpc",        version = "1.69.0")

# ── Non-BCR dep (http_archive via module extension) ────────
non_registry = use_extension("//third_party:extensions.bzl", "non_registry")
use_repo(non_registry, "spdlog")

Bazel Central Registry

The BCR is Bazel's package registry — like npm or crates.io, but for build rules and libraries.


registry.bazel.build

  • Version-pinned modules
  • Checksum-verified downloads
  • Dependency resolution by Bazel

Non-BCR deps

  • Use a module extension
  • http_archive() with explicit SHA256
  • Inject a BUILD file if needed

Non-BCR dep: spdlog

Fetched directly from GitHub via a module extension:


# third_party/extensions.bzl

def _non_registry_deps_impl(ctx):
    http_archive(
        name = "spdlog",
        url  = "https://github.com/gabime/spdlog/archive/refs/tags/v1.14.1.tar.gz",
        sha256 = "1586508029a7d0670dfcb2d97575dcdc242d3868a...",
        strip_prefix = "spdlog-1.14.1",
        build_file = Label("//third_party:spdlog.BUILD"),
    )

non_registry = module_extension(
    implementation = _non_registry_deps_impl,
)

The build_file argument injects our own BUILD rules into the external repo — needed when the library doesn't ship a Bazel build file.

C++ — library, binary, tests

cpp/
  greeter/   ← cc_library (uses abseil from BCR)
  app/      ← cc_binary (uses spdlog, non-BCR)
  server/   ← cc_binary (gRPC server) 


# cpp/greeter/BUILD.bazel
load("@rules_cc//cc:defs.bzl", "cc_library", "cc_test")

cc_library(
    name = "greeter",
    srcs = ["greeter.cc"],
    hdrs = ["greeter.h"],
    deps = ["@abseil-cpp//absl/strings"],   # BCR dep
    visibility = ["//visibility:public"],
)

cc_test(
    name = "greeter_test",
    srcs = ["greeter_test.cc"],
    deps = [":greeter", "@googletest//:gtest_main"],
)

Running C++ targets


# Build
bazel build //cpp/greeter
bazel build //cpp/app:app

# Run
bazel run //cpp/app:app
bazel run //cpp/app:app -- Alice

# Test
bazel test //cpp/greeter:greeter_test --test_output=errors

Labels follow the pattern //package:target
When the target name matches the directory name, you can omit it: //cpp/greeter

Go — same pattern, different rules

Go deps come from go.mod

  • Go packages live on the Go module proxy, not the BCR
  • go.mod at the workspace root
  • Gazelle reads it and generates BUILD files

Workflow


# Add a package
bazel run @rules_go//go -- \
  get github.com/fatih/color@v1.17.0

# Sync use_repo entries
bazel mod tidy

# Regenerate BUILD files
bazel run @gazelle//:gazelle

Go BUILD file


# go/app/BUILD.bazel
load("@rules_go//go:def.bzl", "go_binary")

go_binary(
    name = "app",
    srcs = ["main.go"],
    deps = [
        "//go/greeter",                    # workspace-internal
        "@com_github_fatih_color//:color", # from go.mod via Gazelle
    ],
)

Gazelle generates these files automatically from import statements. The @com_github_fatih_color label is Gazelle's naming convention for Go modules.

Cross-language: Proto / gRPC

One .proto file drives code generation for both C++ and Go.


// proto/greeter.proto
syntax = "proto3";
package greeter;
option go_package = "github.com/mike-boquard/bazel-intro/proto/greeter";

service GreeterService {
  rpc SayHello (HelloRequest) returns (HelloReply);
}

message HelloRequest { string name    = 1; }
message HelloReply   { string message = 1; }

The proto build chain


# proto/BUILD.bazel

# Language-neutral source of truth
proto_library(name = "greeter_proto", srcs = ["greeter.proto"])

# C++ message classes  (greeter.pb.h / .cc)
cc_proto_library(name = "greeter_cc_proto", deps = [":greeter_proto"])

# C++ service stubs  (greeter.grpc.pb.h / .cc)
cc_grpc_library(name = "greeter_cc_grpc",
                srcs = [":greeter_proto"], grpc_only = True,
                deps = [":greeter_cc_proto"])

# Go messages + service stubs in one step
go_proto_library(name = "greeter_go_proto",
                 compilers = ["@rules_go//proto:go_grpc"],
                 importpath = "github.com/mike-boquard/bazel-intro/proto/greeter",
                 proto = ":greeter_proto")
greeter_proto → greeter_cc_proto → greeter_cc_grpc  →  cpp/server
greeter_proto → greeter_go_proto                    →  go/client

C++ server / Go client

cpp/server/BUILD.bazel


cc_binary(
    name = "server",
    srcs = ["main.cc"],
    deps = [
        "//proto:greeter_cc_grpc",
        "//proto:greeter_cc_proto",
        "@grpc//:grpc++",
    ],
)

go/client/BUILD.bazel


go_binary(
    name = "client",
    srcs = ["main.go"],
    pure = "on",
    deps = [
        "//proto:greeter_go_proto",
        "@org_golang_google_grpc//:grpc",
        "@org_golang_google_grpc//credentials/insecure",
    ],
)
        


bazel run //cpp/server:server           # terminal 1: listens on :50051
bazel run //go/client:client -- localhost:50051 Alice   # terminal 2
# → Hello, Alice!

Testing


# Run everything
bazel test //...

# Unit tests
bazel test //cpp/greeter:greeter_test
bazel test //go/greeter:greeter_test

# Cross-language integration test
bazel test //test/integration:greeter_integration_test

Bazel caches test results too — unchanged targets don't re-run. Use --cache_test_results=no to force a re-run.

Integration test as sh_test

A shell script wired together with sh_test — Bazel builds both binaries before running the script.


# test/integration/BUILD.bazel
sh_test(
    name = "greeter_integration_test",
    srcs = ["greeter_test.sh"],
    args = [
        "$(rootpath //cpp/server:server)",   # resolved at test time
        "$(rootpath //go/client:client)",
    ],
    data = [
        "//cpp/server:server",
        "//go/client:client",
    ],
)

$(rootpath ...) expands to the binary's location inside the test sandbox's runfiles tree.

Caching

Local disk cache


# .bazelrc
build --disk_cache=~/.cache/bazel/disk-cache

Content-addressed. If you switch branches and come back, Bazel reuses cached outputs without recompiling.

CI (GitHub Actions)


- uses: actions/cache@v4
  with:
    path: ~/.cache/bazel
    key: ${{ runner.os }}-bazel-${{ hashFiles('MODULE.bazel.lock') }}

The same disk cache directory is restored from the Actions cache. CI shares the same artifact store as local dev.

Querying the build graph

bazel query

Static graph — reads BUILD files, no analysis phase needed.

bazel cquery

Configured graph — understands select(), platforms, and build flags.

bazel aquery

Action graph — shows the actual compiler/linker commands that would run.



# Who uses //cpp/greeter?
bazel query 'rdeps(//..., //cpp/greeter:greeter)'
# → //cpp/app:app  //cpp/greeter:greeter_test

# Why does //cpp/app depend on abseil?
bazel query 'somepath(//cpp/app:app, @abseil-cpp//absl/strings:strings)'
# → //cpp/app:app → //cpp/greeter:greeter → @abseil-cpp//absl/strings:strings

# What compiler flags are used?
bazel aquery 'mnemonic("CppCompile", //cpp/greeter:greeter)' \
  --output=text 2>/dev/null | grep -A1 "Command Line"

Dev ergonomics: clangd support

Bazel uses workspace-rooted includes ("cpp/greeter/greeter.h"), which clangd can't resolve without help.



# Generate compile_commands.json (uses hedron_compile_commands, a non-BCR dep)
bazel run //:refresh_compile_commands
# or:
./scripts/refresh_compile_commands.sh

The file is written to the workspace root, where clangd finds it automatically. Re-run after adding or modifying C++ sources.

Go tooling (gopls) works out of the box via go.mod — no extra step needed.

Takeaways


🔒  Hermetic + incremental — same inputs, same outputs, only rebuild what changed

📦  Bzlmod — declare all deps in MODULE.bazel; BCR or http_archive for the rest

🔀  Polyglot by design — C++, Go, and Proto in one graph, one command

🧪  Testing is first-class — unit tests, integration tests, cross-language sh_test

🔍  Queryable — bazel query/cquery/aquery for understanding the graph

⚡  Cache everything — local disk cache + CI cache, shared transparently
