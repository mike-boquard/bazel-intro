Why Bazel?

You already have make, cmake, go build, cargo…

Hermetic *

Builds don't read from /usr/include or depend on ambient env. Same inputs → same outputs, always.

Incremental

Tracks changes by content hash, not timestamps. Only rebuilds what actually changed — across languages.

Polyglot

C++, Go, Python, Rust, Proto — one build graph, one command. Cross-language dependencies just work.

Correct by construction

The dependency graph must be a DAG. Circular dependencies are a hard error at analysis time — not a cryptic linker failure at build time.

The caching story alone pays off on large codebases. CI and dev machines share the same artifact store.

* C++ is the exception: Bazel defaults to the host compiler (system GCC / Xcode Clang), so compiler version varies across machines. Pin an exact LLVM release with toolchains_llvm to close this gap — see the FAQ slide.

How Bazel works

Persistent JVM server

The bazel CLI is a thin client. The first invocation starts a long-running JVM process that holds the analysis cache in memory. Subsequent builds skip JVM startup and re-analysis — only changed targets rebuild.

Three phases

Loading — reads BUILD.bazel files, evaluates Starlark rules.
Analysis — instantiates the action graph (DAG of commands).
Execution — runs only the actions whose inputs changed.

Sandboxed actions

Every build action and test runs in an isolated environment — only its declared inputs are visible. On Linux: Linux namespaces (like a container). On macOS: sandbox-exec. Undeclared deps are a hard error.


# Log why each target was (re)built or hit cache
bazel build //... --explain=/tmp/build.log

# What actions Bazel will run (the execution plan)
bazel aquery //cpp/app:app

Tests run in the same sandbox — no network, no ambient filesystem. A test that passes locally but fails in CI is almost always an undeclared dependency.

Key vocabulary

Workspace — the repo root; anchored by MODULE.bazel
  └─ Package — any directory containing a BUILD.bazel
     └─ Target — a named buildable unit  //cpp/greeter:greeter_test
         └─ Rule — the type  cc_library / go_binary / sh_test …

Labels follow the pattern //package:target. When the target name matches the directory, you can omit it: //cpp/greeter ≡ //cpp/greeter:greeter

The package file can be named BUILD or BUILD.bazel — both work. If both exist in one directory, BUILD.bazel wins. This repo uses BUILD.bazel throughout for consistency and clearer editor syntax highlighting.

What is Bzlmod?

Bzlmod is Bazel's dependency management system, introduced in Bazel 5, the default since Bazel 6, and the only option in Bazel 9 — the legacy WORKSPACE file was removed entirely. This repo targets Bazel 9.1.1.

WORKSPACE (old)

  • Imperative script — deps fetched in declaration order
  • No real version resolution — you manually manage transitive deps
  • Diamond dependency conflicts are your problem
  • No central registry — every project hand-rolls dep fetching

Bzlmod / MODULE.bazel (new)

  • Declarative — each module states what it needs
  • Bazel resolves the full dep graph, detects conflicts
  • Backed by the Bazel Central Registry (BCR)
  • Lock file (MODULE.bazel.lock) pins exact versions

Think of it as the difference between manually editing node_modules vs using package.json with npm install.

MODULE.bazel — in practice

Every external dependency the repo uses is declared here. bazel_dep is like a package.json entry — versioned and checksum-verified.


module(name = "bazel_intro", version = "0.0.1")

# ── BCR deps (selected) ─────────────────────────────────────────────
bazel_dep(name = "googletest",  version = "1.17.0.bcr.2")
bazel_dep(name = "abseil-cpp",  version = "20250814.1")
bazel_dep(name = "rules_go",    version = "0.61.1")
bazel_dep(name = "protobuf",    version = "33.6")
bazel_dep(name = "grpc",        version = "1.76.0.bcr.1")

# ── Non-BCR deps via module extension ──────────────────────────────
non_registry = use_extension("//third_party:extensions.bzl", "non_registry")
use_repo(non_registry, "spdlog", "hedron_compile_commands")

This repo also declares rules_python, rules_rust, toolchains_llvm, gazelle, and their associated toolchain extensions — the above is a representative subset.

Bazel Central Registry

The BCR (registry.bazel.build) is Bazel's package registry — like npm or crates.io, but for build rules and libraries.

BCR deps

  • Declared with bazel_dep()
  • Version-pinned, checksum-verified
  • Dependency resolution handled by Bazel
  • Lock file: MODULE.bazel.lock

Non-BCR deps

  • Use a module extension
  • http_archive() with explicit URL + SHA256
  • Inject a BUILD file if the library doesn't ship one
  • Example: spdlog, hedron_compile_commands

Non-BCR dep: spdlog

Fetched directly from GitHub via a module extension, with a hand-written BUILD file injected into the downloaded archive:

third_party/extensions.bzl


def _non_registry_deps_impl(ctx):
    http_archive(
        name         = "spdlog",
        url          = "https://github.com/gabime/spdlog/" +
                       "archive/refs/tags/v1.14.1.tar.gz",
        sha256       = "1586508029a7d0670df...",
        strip_prefix = "spdlog-1.14.1",
        build_file   = Label("//third_party:spdlog.BUILD"),
    )

non_registry = module_extension(
    implementation = _non_registry_deps_impl,
)

third_party/spdlog.BUILD


load("@rules_cc//cc:defs.bzl", "cc_library")

cc_library(
    name = "spdlog",
    hdrs = glob(["include/**/*.h"]),
    # Makes #include "spdlog/spdlog.h" resolve.
    includes = ["include"],
    visibility = ["//visibility:public"],
)

build_file points at spdlog.BUILD because spdlog ships no Bazel rules of its own. spdlog is header-only by default (it defines SPDLOG_HEADER_ONLY itself), so no srcs and no defines are needed. The label resolves only because third_party/ is itself a package (it has an empty BUILD.bazel).

C++ — BUILD file


# cpp/greeter/BUILD.bazel
load("@rules_cc//cc:defs.bzl", "cc_library", "cc_test")

cc_library(
    name       = "greeter",
    srcs       = ["greeter.cc"],
    hdrs       = ["greeter.h"],
    deps       = ["@abseil-cpp//absl/strings"],  # BCR dep
    visibility = ["//visibility:public"],
)

cc_test(
    name = "greeter_test",
    srcs = ["greeter_test.cc"],
    deps = [":greeter", "@googletest//:gtest_main"],
)

@abseil-cpp//absl/strings — the @ prefix means an external repo. Labels without @ are internal to this workspace.

C++ — commands


# Build
bazel build //cpp/greeter          # library
bazel build //cpp/app:app          # binary (depends on spdlog)

# Run
bazel run //cpp/app:app
bazel run //cpp/app:app -- Alice   # pass args after --

# Test
bazel test //cpp/greeter:greeter_test
bazel test //cpp/...               # all C++ tests

Bazel downloads the compiler toolchain, abseil, spdlog, and googletest automatically on first build — nothing needs to be installed manually.

Go — same pattern, different rules

Go packages come from go.mod (the Go module proxy), not the BCR. Gazelle reads go.mod and generates BUILD files.


# go/app/BUILD.bazel
load("@rules_go//go:def.bzl", "go_binary")

go_binary(
    name = "app",
    srcs = ["main.go"],
    deps = [
        "//go/greeter",                    # workspace-internal library
        "@com_github_fatih_color//:color", # from go.mod via Gazelle
    ],
)

Gazelle's naming convention: github.com/fatih/color → @com_github_fatih_color.
Workflow: bazel run @rules_go//go -- get pkg@v1.2.3 → bazel mod tidy → bazel run @gazelle//:gazelle
Use bazel run @rules_go//go rather than system go so the same SDK version manages the module.

Cross-language: one .proto, two languages

A single proto/greeter.proto defines the contract for both C++ and Go:


syntax = "proto3";
package greeter;
option go_package = "github.com/mike-boquard/bazel-intro/proto/greeter";

service GreeterService {
  rpc SayHello (HelloRequest) returns (HelloReply);
}

message HelloRequest { string name    = 1; }
message HelloReply   { string message = 1; }
greeter.proto  →  greeter_cc_proto (C++ messages)  →  greeter_cc_grpc (C++ stubs)  →  cpp/server
greeter.proto  →  greeter_go_proto (Go messages + stubs)  →  go/client

Proto build targets

proto/BUILD.bazel turns the single .proto file into four targets — a language-neutral base plus C++ message, C++ service-stub, and Go targets:


# proto/BUILD.bazel — Bazel 8+ requires these rules to be loaded explicitly
load("@grpc//bazel:cc_grpc_library.bzl", "cc_grpc_library")
load("@protobuf//bazel:cc_proto_library.bzl", "cc_proto_library")
load("@protobuf//bazel:proto_library.bzl", "proto_library")
load("@rules_go//proto:def.bzl", "go_proto_library")

proto_library(name = "greeter_proto", srcs = ["greeter.proto"],
              visibility = ["//visibility:public"])

cc_proto_library(name = "greeter_cc_proto", deps = [":greeter_proto"],
                 visibility = ["//visibility:public"])

cc_grpc_library(name = "greeter_cc_grpc", srcs = [":greeter_proto"],
                grpc_only = True, deps = [":greeter_cc_proto"],
                visibility = ["//visibility:public"])

go_proto_library(name = "greeter_go_proto",
                 compilers  = ["@rules_go//proto:go_grpc"],
                 importpath = "github.com/mike-boquard/bazel-intro/proto/greeter",
                 proto      = ":greeter_proto",
                 visibility = ["//visibility:public"])

Integration test

A shell script wired with sh_test. Bazel builds both binaries before running the script — no manual orchestration.


# test/integration/BUILD.bazel
sh_test(
    name = "greeter_integration_test",
    srcs = ["greeter_test.sh"],
    args = [
        "$(rootpath //cpp/server:server)",  # resolved to binary path at test time
        "$(rootpath //go/client:client)",
    ],
    data = [
        "//cpp/server:server",   # listed in data so Bazel builds them first
        "//go/client:client",
    ],
)

The script starts the server, polls nc -z localhost 50051, calls the client, and asserts "Hello, World!".

Testing


bazel test //...                                   # run every test in the workspace
bazel test //cpp/...                               # scope to one subtree
bazel test //cpp/greeter:greeter_test              # one specific target

# Filter to specific test cases (--test_filter passes --gtest_filter / -run)
bazel test //cpp/greeter:greeter_test --test_filter="GreeterTest.HelloWithName"

# Flakiness detection — run each test 10 times
bazel test //... --runs_per_test=10

# Force a re-run even if the result is cached
bazel test //... --nocache_test_results

Results are cached

Unchanged targets don't re-run. Content hash, not timestamp — a git checkout and back is a cache hit.

Output control

--test_output=errors — failing output only (default in .bazelrc)
--test_output=all — always print stdout/stderr
--test_verbose_timeout_warnings — flag tests near their limit

GoogleTest integration

GoogleTest is a BCR dependency — Bazel downloads and builds it from source. No system install, no find_package.

MODULE.bazel


bazel_dep(name = "googletest", version = "1.17.0.bcr.2")

BUILD.bazel


cc_test(
    name = "greeter_test",
    srcs = ["greeter_test.cc"],
    deps = [
        ":greeter",
        # gtest_main supplies main() — no boilerplate needed
        "@googletest//:gtest_main",
    ],
)

greeter_test.cc


#include "gtest/gtest.h"
TEST(GreeterTest, HelloWithName) {
    EXPECT_EQ(greeter::Hello("World"), "Hello, World!");
}

What Bazel does

Each cc_test is compiled into a standalone binary and run in an isolated sandbox. No shared state between tests.

vs CMake / CTest

CMake needs find_package(GTest REQUIRED) — requires GTest on the system. Bazel always builds the exact pinned version from source.

--test_filter

Passes --gtest_filter to the test binary. Run a single test case without rebuilding:

--test_filter="Suite.Case"

Caching: Ninja/CMake vs Bazel

CMake + Ninja Bazel
Cache key File modification time (mtime) SHA-256 of all inputs
After git checkout and back Full rebuild — mtimes changed Cache hit — content identical
Fresh CI clone Always rebuilds everything Hits cache if inputs match
Cross-machine sharing Impossible — mtimes aren't portable Native — remote cache or shared disk
Hermeticity Reads /usr/include, ambient PATH Sandboxed — no ambient environment
CI ↔ dev sharing Not possible CI writes; devs read back

The mtime problem is why "clean builds" are a thing in CMake projects. Bazel doesn't need them.

Caching: in practice

Local disk cache (.bazelrc)


build --disk_cache=~/.cache/bazel/disk-cache

Content-addressed. Switch branches and back — Bazel reuses cached outputs, no recompile.

CI (GitHub Actions)


- uses: actions/cache@v4
  with:
    path: ~/.cache/bazel
    key: ${{ runner.os }}-bazel-${{ hashFiles('MODULE.bazel.lock') }}

CI restores the same directory Bazel writes to locally — artifacts are shared.

Cache key uses MODULE.bazel.lock — adding a dep busts the cache automatically.
For larger teams: Bazel also supports a remote cache (--remote_cache=grpc://...) so every developer hits the CI-populated cache without any local state.

Querying the build graph

bazel query

Static graph. No build needed — reads BUILD files only.

bazel cquery

Configured graph. Understands select(), platforms, build flags.

bazel aquery

Action graph. Shows compiler/linker commands that would actually run.


# Who depends on //cpp/greeter?
bazel query 'rdeps(//..., //cpp/greeter:greeter)'

# Why does //cpp/app pull in abseil?
bazel query 'somepath(//cpp/app:app, @abseil-cpp//absl/strings:strings)'

# What compiler flags are used?
bazel aquery 'mnemonic("CppCompile", //cpp/greeter:greeter)' \
  --output=text 2>/dev/null | grep -A1 "Command Line"

Hermetic Python with rules_python

The same hermetic story applies to Python: pinned interpreter + pinned packages with content hashes.

MODULE.bazel


bazel_dep(name = "rules_python", version = "2.0.3")

python = use_extension(
    "@rules_python//python/extensions:python.bzl", "python")
python.toolchain(python_version = "3.12")  # hermetic interpreter
use_repo(python, "python_3_12", "pythons_hub")

    "@rules_python//python/extensions:pip.bzl", "pip")

          requirements_lock = "//:requirements_lock.txt")
use_repo(pip, "pypi")

requirements_lock.txt (excerpt)


rich==14.0.0 \
  --hash=sha256:1c9491e1... \
  --hash=sha256:82f1bc23...

  --hash=sha256:6757cd03...

BUILD.bazel

Rust with rules_rust

Same hermetic story: pinned rustc + crates from crates.io via crate_universe. No system Rust required.

MODULE.bazel


bazel_dep(name = "rules_rust", version = "0.70.0")

rust = use_extension(
    "@rules_rust//rust:extensions.bzl", "rust")
rust.toolchain(edition = "2021", versions = ["1.85.0"])
use_repo(rust, "rust_toolchains")
register_toolchains("@rust_toolchains//:all")

# crates.io deps — analogous to go_deps for Go
crate = use_extension(
    "@rules_rust//crate_universe:extensions.bzl", "crate")
crate.spec(package = "colored", version = "2")
crate.from_specs()
use_repo(crate, "crates")

rust/greeter/src/lib.rs — inline tests (idiomatic)


pub fn hello(name: &str) -> String {
    format!("Hello, {}!", name)
}

#[cfg(test)]
mod tests {
    use super::*;
    #[test]
    fn hello_with_name() {
        assert_eq!(hello("World"), "Hello, World!");
    }
}

BUILD.bazel


rust_library(name = "greeter", srcs = ["src/lib.rs"],
             edition = "2021", visibility = ["//visibility:public"])

# runs the #[cfg(test)] module inside the library crate
rust_test(name = "greeter_test", crate = ":greeter")

bazel run //rust/app:app ·  bazel test //rust/greeter:greeter_test

Prefer a real Cargo.toml? Swap crate.from_specs() for crate.from_cargo(manifests = ["//:Cargo.toml"], cargo_lockfile = "//:Cargo.lock") — crate_universe reads your existing manifest + lockfile instead of inline specs. This repo uses from_specs() to keep everything in MODULE.bazel with no extra Cargo files.

Takeaways

🔒  Hermetic + incremental — same inputs always produce the same outputs

📦  Bzlmod — all deps in MODULE.bazel; BCR or http_archive for the rest

🔀  Polyglot — C++, Go, Python, Rust, and Proto in one graph, one bazel build //...

🧪  Testing first-class — unit tests, integration tests, cross-language sh_test

⚡  Cache everything — local disk + CI, shared by content hash

🔍  Queryable graph — query / cquery / aquery to inspect any dependency

FAQ

Common questions from teams adopting Bazel

FAQ: Can I pin a specific Clang version?

Yes — toolchains_llvm from the BCR downloads an exact pre-built LLVM release. This repo wires up LLVM 18.1.8 as an opt-in toolchain for reproducible Linux x86_64 builds (--config=clang); the default build (and CI) uses the host compiler. macOS uses Xcode Clang (pre-built arm64 binaries top out at LLVM 17 on Apple Silicon).


# MODULE.bazel
bazel_dep(name = "toolchains_llvm", version = "1.8.0")

llvm = use_extension("@toolchains_llvm//toolchain/extensions:llvm.bzl", "llvm")
llvm.toolchain(llvm_version = "18.1.8")   # pinned LLVM release
use_repo(llvm, "llvm_toolchain")
# No register_toolchains() — opt in per-platform instead (see .bazelrc)
  

# .bazelrc — opt-in pinned-LLVM config (Linux x86_64)
build:clang --extra_toolchains=@llvm_toolchain//:cc-toolchain-x86_64-linux
  

bazel build --config=clang //cpp/...   # opt in to pinned LLVM 18.1.8 (Linux)
bazel build //cpp/...                  # default: host compiler (gcc / Xcode Clang)

FAQ: Code formatting with clang-format

The same toolchains_llvm download that provides the compiler also exposes clang-format 18.1.8 as a first-class Bazel target — no system install needed, same binary on every machine and in CI.


# Format all C++ files in-place
./scripts/format_cpp.sh

# Check mode — exits 1 if any file would change (useful in CI)
./scripts/format_cpp.sh --check
  

# Or invoke the target directly with an absolute path
bazel run //:clang-format -- -i /abs/path/to/file.cc

Style is declared in .clang-format at the workspace root:


BasedOnStyle: LLVM
IndentWidth: 4
ColumnLimit: 100

Note: the script passes absolute paths because bazel run may resolve the binary's working directory relative to its runfiles tree rather than the workspace root. The //:clang-format alias in BUILD.bazel is a convenience pointer to @llvm_toolchain//:clang-format.

FAQ: Custom flags & sanitizers

Three scopes for compiler flags:

Per-target

copts = ["-O3"] in cc_library / cc_binary

Per-invocation

bazel build --copt=-march=native

Persistent (.bazelrc)

build --cxxopt=-std=c++17 — applied to every build

Scope flags to your own code — --cxxopt also hits external deps (abseil, grpc…), which emit warnings you can't fix. --per_file_copt matches a regex on the source path, so warnings-as-errors apply to first-party files only:


# .bazelrc — -Wall -Werror for our sources under cpp/, not external deps
build --per_file_copt=^cpp/.*\.cc$@-Wall,-Werror

That scopes where the flags apply — but a first-party file that #includes a third-party header still sees that header's warnings. Route external headers through -isystem so they don't count under -Werror:


build --features=external_include_paths

Sanitizers — add a named config to .bazelrc:


build:asan  --copt=-fsanitize=address   --linkopt=-fsanitize=address
build:tsan  --copt=-fsanitize=thread    --linkopt=-fsanitize=thread
build:ubsan --copt=-fsanitize=undefined --linkopt=-fsanitize=undefined
  

bazel test --config=asan //...

FAQ: Cross-compilation

Bazel has first-class platform support. Define a platform target and select it at build time:


# platforms/BUILD.bazel
platform(
    name = "linux_arm64",
    constraint_values = [
        "@platforms//os:linux",
        "@platforms//cpu:arm64",
    ],
)
  

bazel build --platforms=//platforms:linux_arm64 //cpp/app:app

toolchains_llvm supports cross-compilation out of the box — configure it with exec_arch and target_arch. This is generally cleaner than CMake toolchain files because the platform is a first-class build graph node, not an external file sourced before analysis.

FAQ: Migrating from CMake

Incremental migration

CMakeLists.txt and BUILD.bazel can coexist in the same repo indefinitely. Migrate package by package; run both build systems until confidence is high.

CMake-only dependencies

rules_foreign_cc wraps a CMake build as a hermetic Bazel action. Bazel invokes CMake internally and treats the outputs as normal build artifacts.


# Wrap a CMake dep inside Bazel
load("@rules_foreign_cc//foreign_cc:defs.bzl", "cmake")

cmake(
    name  = "libfoo",
    lib_source = "@libfoo_src//:all_srcs",
    out_static_libs = ["libfoo.a"],
)

Go migration is easier — Gazelle generates BUILD files from go.mod automatically. C++ requires more manual work per package.

FAQ: System libraries & vendored deps

System libraries (OpenSSL, zlib, libcurl…) — three options, in order of preference:

1. BCR

Check first — boringssl, zlib, and others are already there.

2. http_archive

Download from source, provide a BUILD file. Fully hermetic.

3. rules_foreign_cc

Build their existing CMake/autoconf build inside Bazel.

Vendored dependencies — wrap a local directory in a module extension (Bzlmod requires these calls inside an extension function, not bare in MODULE.bazel):


# third_party/extensions.bzl — inside a module extension function
local_repository(name = "mylib", path = "vendor/mylib")

# Or use local_path_override to redirect a BCR module to a local fork
local_path_override(module_name = "mylib", path = "vendor/mylib")

FAQ: Go specifics

go build / go test

Still work — go.mod is a normal Go module. You lose the Bazel cache when using them, but nothing breaks. Teams often run both during adoption.

cgo

Supported — requires a C toolchain to be configured. Use pure = "on" on a go_binary to opt out explicitly. Works, but adds complexity.

go:generate

No native equivalent. Bazel wants all codegen as explicit rules. Wrap output in a genrule, or commit generated files and document the generation step separately.

Gazelle handles the common case well — it reads import statements and generates BUILD files automatically. Edge cases (build tags, internal packages) require manual tuning.

FAQ: Performance & scale

Cold build vs CMake+Ninja

First build: comparable or slightly slower (Bazel also fetches deps). Subsequent builds: faster, because content-hash caching is exact. At scale — large teams, many targets — the gap becomes dramatic.

Remote execution (RBE)

Bazel can distribute actions across a build cluster (--remote_executor=grpc://...). Remote cache (--remote_cache=) is a simpler first step — CI writes, every developer reads. Open-source options: BuildBuddy, NativeLink, EngFlow.

Parallelism: --jobs=N controls concurrency (default = CPU count). Bazel schedules more aggressively than Ninja because it holds the full dependency graph and knows exactly which actions are independent.

FAQ: Honest rough edges

Learning curve

The hermetic mindset is a genuine shift. "Why can't I read /usr/local/include?" is a common early frustration. Most teams take a few weeks to feel fluent.

Error messages

Toolchain resolution failures in particular can be cryptic. --toolchain_resolution_debug=.* helps but produces walls of output.

Bazel version upgrades

Bumping .bazelversion can break builds. The BCR compatibility matrix and Bzlmod mitigate this, but it's not frictionless. Pin and upgrade deliberately.

These are real. Worth setting expectations with the team upfront rather than discovering them in week two.
